# Attack reproduction — DGA replay

Real DGA malware is hard to acquire safely; the most common lab approach
is to *replay* a public DGA family's daily domain list and resolve it
from a victim host. Zeek records the DNS-query burst that fires the
detection in `../detection/`.

## Prereqs

- Lab compose stack running so Zeek + Splunk are capturing
- Victim host on the lab subnet

## Run (high level — replay a known DGA family)

```bash
# Pull a list of known DGA domains (e.g., DGArchive, Bambenek, Netlab 360)
curl -fsSL https://data.netlab.360.com/feeds/dga/dga.txt \
  | awk '$1 == "Conficker" {print $2}' \
  | head -50 \
  > /tmp/dga_today.txt

# Resolve each — most will NXDOMAIN (operator hasn't registered them this week)
while read -r d; do dig +short "$d" A >/dev/null 2>&1 ; done < /tmp/dga_today.txt
```

## Capture telemetry into a fixture

```bash
docker exec detlab-zeek cat /usr/local/zeek/logs/current/dns.log \
  | jq -c "select(.\"id.orig_h\" == \"<victim_ip>\")" \
  > /tmp/dga_capture.log
head -100 /tmp/dga_capture.log > tests/positive_dns.log
```

## Safety / scope

- Lab use only.
- Resolving DGA domains is harmless — most NXDOMAIN. Live ones may be
  sinkholes or active C2; don't follow up with HTTP requests.
- The synthetic `tests/positive_dns.log` shipped here is generated by
  `scripts/generate_fixtures.py` so the test suite is hermetic.
